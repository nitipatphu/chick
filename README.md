## 🎯 วัตถุประสงค์
ทำโปรเจกต์นี้ขึ้นมาเพื่อ:
1. สร้างระบบขายอุปกรณ์กีฬาออนไลน์ที่ใช้งานได้จริง
2. ให้ลูกค้าเข้ามาค้นหาสินค้าและกดสั่งซื้อได้แบบสะดวกๆ
3. ช่วยให้แอดมินและพนักงานมีระบบหลังบ้านที่จัดการสต็อกและออเดอร์ได้ง่ายขึ้น

---

## 📌 หลักการและเหตุผล
เดี๋ยวนี้ใครๆ ก็ซื้อของออนไลน์กันทั้งนั้น โดยเฉพาะอุปกรณ์กีฬาที่มีให้เลือกเยอะมาก พวกเราเลยอยากทำระบบที่ช่วยอำนวยความสะดวกให้ลูกค้าเข้ามาดูของและกดสั่งซื้อได้เลย ส่วนหลังบ้านก็ออกแบบมาให้คนขายจัดการข้อมูลออเดอร์และสต็อกสินค้าได้แบบไม่ปวดหัวครับ

---

## 👤 ขอบเขตระบบ (System Scope)

### ผู้ที่เกี่ยวข้องในระบบ (Actors)

| Actor | บทบาท | ทำอะไรได้บ้าง |
|-------|-------|-----------|
| **Customer** | ลูกค้าทั่วไป | - ค้นหาและเข้ามาดูสินค้า<br>- กดหยิบของใส่ตะกร้าและสั่งซื้อ<br>- อัปโหลดสลิปโอนเงิน<br>- เช็คประวัติการสั่งซื้อของตัวเอง<br>- แก้ไขโปรไฟล์ส่วนตัว |
| **Staff** | พนักงาน | - ล็อกอินด้วยสิทธิ์พนักงาน<br>- เช็คและอัปเดตสถานะออเดอร์ (เช่น แพ็คของแล้ว, ส่งแล้ว)<br>- อัปเดตเลขพัสดุ (Tracking)<br>- เช็คสต็อกสินค้า |
| **Administrator** | ผู้ดูแลระบบ | - จัดการข้อมูลสินค้าได้ทั้งหมด (เพิ่ม/ลบ/แก้ไข)<br>- อนุมัติสลิปการโอนเงิน<br>- จัดการข้อมูลผู้ใช้งานในระบบ<br>- ดูรายงานและสถิติยอดขาย |

---

### ความสามารถหลักของระบบแบ่งตามผู้ใช้งาน

#### 🛒 สำหรับลูกค้า (Customer)
- สมัครสมาชิกและเข้าสู่ระบบ
- ค้นหาสินค้าตามหมวดหมู่
- ดูรายละเอียดสินค้า (ไซส์, สี, ราคา)
- เพิ่มสินค้าลงตะกร้าและกดสั่งซื้อ
- แนบสลิปชำระเงิน
- ตรวจสอบประวัติและสถานะคำสั่งซื้อ

#### 📦 สำหรับพนักงาน (Staff)
- เข้าสู่ระบบด้วยสิทธิ์พนักงาน
- จัดการคำสั่งซื้อ (เปลี่ยนสถานะเป็นจัดส่งแล้ว)
- กรอกเลขพัสดุให้ลูกค้า
- ตรวจสอบสินค้าคงคลัง

#### ⚙️ สำหรับแอดมิน (Administrator)
- มีสิทธิ์ทำได้ทุกอย่างในระบบ
- จัดการข้อมูลสินค้าและหมวดหมู่ทั้งหมด
- ตรวจสอบและกดยืนยันสลิปโอนเงิน
- จัดการข้อมูลสมาชิกในระบบ
- ดูรายงานและสถิติยอดขาย

## 📊 ผลลัพธ์ที่คาดว่าจะได้รับ
1. ได้เว็บขายอุปกรณ์กีฬาที่เอาไปใช้งานได้จริง
2. ลูกค้า (Customer) สามารถกดสั่งซื้อของผ่านเว็บได้สะดวกและรวดเร็ว
3. พนักงาน (Staff) ทำงานง่ายขึ้น จัดการออเดอร์และสต็อกได้เป็นระบบ
4. แอดมิน (Administrator) ควบคุมทุกอย่างในร้านได้จากที่เดียว
5. ลดความผิดพลาดเรื่องออเดอร์ตกหล่นและสต็อกสินค้าไม่ตรง

## 📅 แผนการดำเนินงาน (4 สัปดาห์)

| สัปดาห์ | กิจกรรม | สิ่งที่ทำ |
|---------|---------|-----------|
| **สัปดาห์ที่ 1** | วางแผนและวิเคราะห์ (Analysis & Design) | - นั่งคุยความต้องการของระบบและตกลงขอบเขตการทำงาน |
| **สัปดาห์ที่ 2** | ออกแบบและพัฒนาหน้าเว็บ (Frontend) | - ปั้น UI/UX ให้สวยงามและเขียนหน้าเว็บด้วย React |
| **สัปดาห์ที่ 3** | ทำระบบหลังบ้าน (Backend Development) | - เขียน API ด้วย Node.js<br>- เชื่อมข้อมูลเข้ากับไฟล์ `db.json`<br>- เอาหน้าเว็บมาต่อกับหลังบ้าน |
| **สัปดาห์ที่ 4** | ทดสอบและปรับปรุง (Testing & Deployment) | - ลองกดเทสระบบ หาบั๊กและแก้ไข<br>- รันเทสด้วย Docker<br>- จัดทำเอกสารส่งงาน |

## 🔄 แนวทางการพัฒนา (ตามหลัก SDLC)

| ขั้นตอน | รายละเอียด | สิ่งที่ทำในขั้นตอนนี้ |
|---------|-----------|-----------|
| **1. Planning** | วางแผนโครงการ | - กำหนดขอบเขตว่าโปรเจกต์นี้จะทำอะไรบ้าง |
| **2. Analysis** | วิเคราะห์ความต้องการ | - วิเคราะห์ว่าระบบต้องมีฟังก์ชันอะไรเพื่อตอบโจทย์ผู้ใช้งาน |
| **3. Design** | ออกแบบระบบ | - ออกแบบฐานข้อมูล (Class Diagram)<br>- ออกแบบหน้าตาเว็บ (UI/UX) |
| **4. Development** | เริ่มเขียนโค้ด | - หน้าบ้านใช้ React<br>- หลังบ้านใช้ Node.js<br>- เชื่อมต่อหน้าบ้านและหลังบ้านเข้าด้วยกัน |
| **5. Testing** | ทดสอบระบบ | - ลองใช้งานจริง (UAT) เพื่อหาบั๊ก |
| **6. Deployment** | นำไปใช้งานจริง | - รันระบบผ่าน Docker <br>- เตรียมส่งมอบงาน |
| **7. Maintenance** | ดูแลรักษา | - คอยแก้บั๊กที่อาจจะเจอทีหลัง<br>- ดูแลระบบให้รันได้ลื่นไหล |

## 🧪 การทดสอบระบบ (Testing)

### 1. การทดสอบที่เราทำ

| ประเภท | รายละเอียด |
|--------|-----------|
| **Functional Testing** | ลองกดเล่นดูว่าระบบทำงานถูกไหม (เช่น สั่งของแล้วสต็อกลดหรือเปล่า) |
| **User Acceptance Testing (UAT)** | ลองให้คนอื่นมาใช้งานจริงว่าใช้งานง่ายไหม |

---

### 2. เครื่องมือที่เอามาช่วยเทส

| เครื่องมือ | เอาไว้ทำอะไร |
|-----------|----------|
| **Postman** | เอาไว้ยิงเทส API ของระบบหลังบ้าน |
| **Browser DevTools** | เอาไว้เช็คความผิดปกติหรือบั๊กบนหน้าเว็บ |
| **Docker** | เอาไว้จำลองสภาพแวดล้อมตอนรันระบบจริง |
